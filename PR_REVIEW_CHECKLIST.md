# PR 作业审核清单（按 Week 要求）

> 用途：按照各周 README 的任务要求，审核同学 PR，并在对应 PR 评论区给出「优点 / 不足 / 优化方向」。

---

## 一、审核流程（建议）

1. 确认 PR 对应周次（`week1` ~ `week6`）与目录位置是否正确。  
2. 对照该周 `README` 的“必做项 / 验收清单”逐条核对。  
3. 阅读代码、笔记、部署信息（若该周要求）。  
4. 在 PR 评论区使用下方模板输出：
   - 优点（至少 2 点）
   - 不足（具体到功能、质量或规范）
   - 优化方向（可执行、可落地）

---

## 二、Week 对应审核重点

### Week1（`/week1/README.md`）
- ERC20 基础实现是否完整（如 transfer、approve、transferFrom 等核心逻辑）
- 是否有部署信息或学习记录
- 代码结构、命名、基础安全意识（权限、输入检查）

### Week2（`/week2/README.md`）
- 升级版 ERC20：是否基于 OpenZeppelin 思路实现扩展（mint / burn / maxSupply / 事件）
- Voting 合约：候选人、投票限制（同地址/NFT 一次）、查询接口、事件是否完整
- 是否包含部署信息、问题总结与学习笔记

### Week3（`/week3/readme.md`）
- 是否提交 Uniswap V2 学习笔记（核心机制、手续费、数据流）
- 是否包含 Week4 设计草稿（模块拆分、关键状态、函数流程）
- 学习总结是否体现“理解”而非只做摘抄

### Week4（`/week4/README.md`）
- 是否手写 Factory + Pair（非直接复制官方）
- `mint/burn/swap` 与 `x*y=k`、0.3% fee 逻辑是否合理
- 是否提交与官方实现的对齐分析（差异、取舍、风险点）

### Week5（`/week5/README.md`）
- 是否实现最小 Swap UI（输入、自动计算、方向切换）
- 前端结构是否清晰（HTML/CSS/JS）
- README 是否包含截图、逻辑说明、部署地址

### Week6（`/week6/README.md`）
- 是否完成 Remix 部署（Factory / Pair）
- 是否有前端可连接钱包、读取 reserve、发起交易
- `deployment.md` / README 是否包含地址与关键交易哈希

## 三、PR 评论模板（可直接复制）

```md
### ✅ 作业评审反馈（WeekX）

#### 1) 优点
- （示例）目录结构清晰，提交内容与 WeekX 任务要求高度匹配。
- （示例）核心功能实现完整，关键流程可读性较好。
- （示例）文档较完整，能体现学习过程与问题排查。

#### 2) 不足
- （示例）部分边界条件处理不足，例如：____。
- （示例）事件/权限/输入校验不够完整，例如：____。
- （示例）README 中缺少部署信息或关键说明：____。

#### 3) 优化方向
1. 先补齐验收清单中的缺失项（按本周 README 的“必做项”逐条对齐）。
2. 针对合约补充安全与可维护性细节（输入校验、事件、权限边界）。
3. 在 README 增加“问题-原因-解决”小结，形成可复用经验。

---
整体评价：**（通过 / 建议修改后通过）**
```

---

## 四、快速判定标准（建议）

- **通过**：本周必做项基本完成，且无明显关键缺陷。  
- **建议修改后通过**：主功能完成，但存在缺失项或质量问题。  
- **需继续完善**：与本周任务目标偏差较大，需补齐核心交付物。
